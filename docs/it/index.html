Perché antivirus.sh

La maggior parte delle macchine Linux sono server che vengono compromessi sempre dalle stesse banali porte d'ingresso: porte esposte, SSH con password, cron job dimenticati, pacchetti non aggiornati. antivirus.sh chiude quelle porte in pochi minuti — e va a caccia di chi è già entrato.

📦

Un solo file, zero dipendenze

Un unico script Bash. Niente Python, niente agent, niente demoni, niente da compilare. Scaricalo, eseguilo, cancellalo se vuoi — ti lascia un report e (facoltativamente) un sistema hardened.

🐧

Qualsiasi Ubuntu. Qualsiasi Linux.

Garantito su ogni release di Ubuntu — dalle LTS più datate all'ultima uscita. Supporto best-effort per Debian, RHEL, CentOS, Alma, Rocky, Fedora, Arch, openSUSE, Alpine: ogni controllo degrada in modo controllato.

🔍

Trova il malware del mondo reale

Crypto-miner, reverse shell, dropper nei cron, backdoor via chiavi SSH, rootkit LD_PRELOAD, payload fileless memfd, processi nascosti — le cose che si trovano davvero sui server violati.

🧯

Fix, non solo segnalazioni

Ogni problema arriva con un fix applicabile: in modo interattivo, automaticamente con --fix, oppure come setup completo di una nuova VM con --harden. O resta al 100% in sola lettura con --audit.

🪢

Non ti chiude fuori

L'attivazione del firewall autorizza prima le tue porte SSH. L'SSH a sola chiave viene rifiutato finché una chiave non esiste davvero. Le nuove configurazioni di sshd vengono validate prima del reload. Tutto viene salvato in backup ed è reversibile con --rollback.

🤝

Va d'accordo con i pezzi grossi

Rileva e orchestra ClamAV, rkhunter e chkrootkit quando sono presenti — un solo flag li installa. Usa antivirus.sh come prima linea rapida e come collante attorno agli scanner approfonditi.

Il catalogo completo dei controlli

Più di 70 controlli individuali in sei aree. Ogni risultato è classificato OK / INFO / WARN / CRIT e contribuisce a un punteggio di sicurezza finale da 0 a 100.

01 Malware e rootkit

  • nomi di processo di miner e malware noti (xmrig, kinsing, kdevtmpfsi…) e righe di comando da mining pool
  • processi in esecuzione da /tmp, /var/tmp, /dev/shm
  • processi il cui binario è stato eliminato; eseguibili fileless memfd
  • rilevamento di processi nascosti (test anti kernel-rootkit basato sull'occultamento in readdir)
  • moduli kernel corrispondenti a rootkit noti (Diamorphine, Reptile…)
  • rootkit userland via /etc/ld.so.preload e LD_PRELOAD globale
  • eseguibili nascosti nelle directory temporanee e in /dev
  • pattern di reverse shell / dropper / miner negli script fino a 2 MB
  • integrità dei binari fondamentali rispetto ai checksum dei pacchetti (dpkg -V / rpm -V)
  • orchestrazione di ClamAV, rkhunter, chkrootkit (opzionale)

02 Persistenza e backdoor

  • crontab di sistema e degli utenti, cron.d, hourly/daily/weekly/monthly, job at
  • unit systemd personalizzate eseguite da directory temporanee o corrispondenti a pattern d'attacco
  • rc.local, script init, regole udev, script MOTD
  • file di avvio della shell di ogni utente (.bashrc, .profile, .zshrc…)
  • audit di authorized_keys per ogni account — segnala le chiavi SSH sugli account di sistema
  • account UID-0 aggiuntivi, regole sudo NOPASSWD, vecchi file di trust .rhosts
  • revisione dei repository APT di terze parti

03 Esposizione di rete

  • ogni porta TCP/UDP in ascolto con processo proprietario e indirizzo di bind
  • classificazione dell'esposizione: solo loopback vs raggiungibile dal mondo
  • servizi pericolosi segnalati: Telnet, FTP, r-services, SMB, RDP, VNC, rsync, NFS, rpcbind
  • datastore senza autenticazione di default: Redis, MongoDB, Elasticsearch, memcached
  • rilevamento della Docker API su :2375 (root remoto istantaneo)
  • connessioni stabilite verso porte di mining pool / botnet IRC
  • interfacce in modalità promiscua, indicatori di ARP spoofing
  • revisione dei resolver DNS e rilevamento di hijack di /etc/hosts

04 SSH e account

  • login root, autenticazione con password, password vuote, X11 forwarding
  • MaxAuthTries, login grace, keep-alive, autenticazione host-based, rhosts
  • presenza di fail2ban con configurazione automatica di una jail sshd
  • tracce di brute force nei record di login falliti di btmp
  • account con password vuota (con blocco immediato), UID duplicati
  • account di sistema con shell di login, revisione dell'appartenenza al gruppo sudo
  • policy di scadenza delle password e robustezza imposta con pam_pwquality

05 Kernel e filesystem

  • 16 chiavi sysctl di hardening: ASLR, syncookies, redirect, source routing, rp_filter, ptrace scope, restrizioni dmesg/kptr, protected links…
  • permessi su shadow, sudoers, chiavi host SSH, crontab, configurazione di GRUB
  • audit SUID/SGID rispetto a una whitelist curata + prova di appartenenza ai pacchetti
  • file scrivibili da chiunque e directory world-writable senza sticky bit
  • file senza proprietario, lock da malware con flag immutable (chattr +i)
  • flag di mount di /dev/shm e /tmp (noexec, nosuid, nodev)
  • restrizioni sui core dump e prevenzione dei dump SUID

06 Piattaforma e servizi

  • aggiornamenti di sicurezza in sospeso, unattended-upgrades, rilevamento di release EOL
  • rilevamento del riavvio richiesto dopo le patch del kernel
  • stato del mandatory access control AppArmor / SELinux
  • traccia forense di auditd, logging journald persistente
  • sincronizzazione oraria NTP (TLS e analisi forense ne dipendono)
  • mitigazioni delle vulnerabilità della CPU (famiglia Meltdown/Spectre)
  • Docker: permessi del socket, container privilegiati, membri del gruppo docker
  • stato del Secure Boot, raccomandazione di una password per GRUB

Installazione e prima esecuzione

Nessuna installazione nel senso classico — è solo uno script. Tre comandi e hai un report completo.

download
curl -fsSL https://raw.githubusercontent.com/TARGET_PLEVEHOLDER/antivirus.sh/main/antivirus.sh -o antivirus.sh
1 — audit sicuro in sola lettura (non modifica nulla)
sudo bash antivirus.sh --audit
2 — interattivo: esamina e conferma ogni fix
sudo bash antivirus.sh
3 — oppure hardening completo di una nuova VM
sudo bash antivirus.sh --harden

Suggerimento: funziona anche senza root — eseguirà ogni controllo possibile e segnalerà chiaramente il resto. funziona anche con git clone: git clone https://github.com/TARGET_PLEVEHOLDER/antivirus.sh

Modalità e opzioni

Uno script, quattro modi di usarlo: audit, interattivo, automatico, hardening completo. Più gli ambiti mirati per scripting e cron.

ComandoCosa succede
sudo bash antivirus.shModalità interattiva — ogni risultato viene mostrato, ogni fix confermato singolarmente. Consigliata per la prima esecuzione.
--auditSolo report. Zero modifiche garantite — sicuro in produzione. Exit code 0/1/2 = pulito/warning/critico, perfetto per cron e CI.
--fixApplica automaticamente tutti i fix sicuri. I fix rischiosi (firewall, autenticazione SSH) richiedono comunque una conferma esplicita.
--hardenTutto ciò che fa --fix, più il kit completo per una nuova VM: firewall, fail2ban, unattended-upgrades, set sysctl, auditd, policy delle password, banner, lockdown di core dump e shm, e la proposta di creare un vero utente amministratore.
--create-userCreazione guidata di un utente sudo: password robusta generata, installazione della chiave SSH, passaggio opzionale all'SSH a sola chiave — con il promemoria di fare un test prima di chiudere la sessione.
--scan /var/wwwScansione malware di una directory specifica (le web root sono un nascondiglio prediletto).
--network / --system / --malwareEsegue una sola area.
--quickSalta le parti lente (scansioni dell'intero disco, ClamAV) — termina in pochi secondi.
--fullLa modalità più approfondita: scansione dell'intero filesystem, verifica di ogni pacchetto installato.
--rollbackRipristina ogni file modificato nell'ultima esecuzione e annulla l'attivazione del firewall.
--install-toolsInstalla ClamAV, rkhunter e chkrootkit per la profondità basata su firme.
--report file.txt · --no-color · --yes · --exclude PATHStrumenti di supporto per reportistica e automazione.

Progettato perché tu non possa chiuderti fuori

Gli strumenti di hardening hanno la brutta abitudine di tagliare il ramo su cui sei seduto. antivirus.sh tratta l'accesso remoto come sacro:

  • L'attivazione del firewall autorizza prima SSH. Le tue porte SSH vengono rilevate in tre modi (configurazione di sshd, socket attivi, connessione corrente) e autorizzate prima che venga eseguito ufw enable.
  • L'SSH a sola chiave viene rifiutato senza una chiave. PasswordAuthentication no viene applicato solo se un utente con privilegi sudo ha davvero una voce in authorized_keys. Lo stesso vale per la disattivazione del login root.
  • Le configurazioni di sshd vengono validate prima del reload. Ogni modifica passa da sshd -t; una configurazione non valida viene ripristinata automaticamente. Le sessioni SSH attive non vengono mai interrotte.
  • Tutto viene salvato in backup. Ogni file modificato viene copiato in una directory di backup con timestamp e manifest; --rollback ripristina tutto, inclusa la disattivazione di un firewall attivato dallo script.
  • Quarantena, mai eliminazione. I file sospetti vengono spostati in una directory di quarantena e privati di ogni permesso — prove e reversibilità preservate.
  • I fix rischiosi chiedono sempre. Anche in modalità --fix, qualsiasi cosa possa toccare l'accesso remoto richiede una conferma esplicita (oppure il documentato e deliberato --yes).

Ricetta: VM nuova di zecca in 3 minuti

Hai appena ricevuto una VM Ubuntu fresca dal tuo cloud provider, password di root alla mano. Ecco l'intera routine:

passo 1 — scarica ed esegui l'hardening
curl -fsSL https://raw.githubusercontent.com/TARGET_PLEVEHOLDER/antivirus.sh/main/antivirus.sh -o antivirus.sh
sudo bash antivirus.sh --harden

Durante l'esecuzione lo script: attiva il firewall con SSH già autorizzato → installa fail2ban e unattended-upgrades → applica 16 chiavi di hardening del kernel → irrobustisce SSH (validato, senza perdere la sessione) → imposta la policy delle password → blinda /dev/shm, i core dump e le shell inattive → propone di creare il tuo utente sudo personale con chiave SSH → e solo allora, con la tua chiave al suo posto, propone di passare all'SSH a sola chiave e disattivare il login root. Prova il nuovo accesso in un secondo terminale — la tua sessione corrente resta attiva per tutto il tempo.

FAQ

È un antivirus "vero" come ClamAV?

È un livello diverso. ClamAV confronta milioni di firme di file; antivirus.sh dà la caccia ai pattern che compromettono davvero i server Linux — miner, reverse shell, dropper nei cron, backdoor SSH, tracce di rootkit — e fa l'audit della configurazione che li ha lasciati entrare. Si completano a vicenda, ed è per questo che antivirus.sh rileva ClamAV, lo usa quando è presente e te lo installa con --install-tools.

Funzionerà sulla mia vetusta Ubuntu 14.04 / sull'ultimissima release?

Sì — è un vincolo di progetto. Puro Bash, nessuna parte compilata, ogni controllo ha dei fallback (ss/netstat, systemd/sysvinit, ufw/iptables) e viene saltato con garbo, con una nota, quando una funzionalità non esiste. Lo stesso script gira dai sistemi dell'era Ubuntu 12.04 fino all'ultima release.

È sicuro eseguirlo in produzione?

La modalità --audit è rigorosamente in sola lettura ed è la prima esecuzione consigliata ovunque. Le modalità che modificano qualcosa confermano ogni passaggio (interattiva) o si limitano ai fix sicuri (--fix), mantengono backup di ogni file toccato e mettono a disposizione --rollback.

Davvero non può chiudermi fuori da SSH?

Le transizioni pericolose — attivare un firewall, disattivare l'autenticazione con password, disattivare il login root — sono protette: le porte SSH vengono autorizzate in anticipo, la presenza della chiave viene verificata prima della modalità a sola chiave, le configurazioni vengono validate con sshd -t e ripristinate automaticamente in caso di errore, e la tua sessione corrente non viene mai interrotta. Ti consigliamo comunque di provare un nuovo accesso in un secondo terminale; lo script te lo ricorda esattamente al momento giusto.

Ha trovato qualcosa di CRITICO. E adesso?

Leggi la segnalazione — ognuna si spiega da sola e stampa le prove. Per le segnalazioni di malware, metti in quarantena con il fix proposto, poi chiediti come è entrato (servizio esposto? chiave rubata? CMS datato?). Importante: se ci sono indizi di un rootkit a livello kernel (processi nascosti, moduli rootkit), nessuna pulizia sul sistema live è affidabile — ricostruisci da un'immagine pulita e ripristina soltanto i dati.

Telefona a casa o carica qualcosa?

No. Nessuna telemetria, nessun upload, nessuna chiamata esterna tranne quelle che richiedi tu (installazione di pacchetti, aggiornamento delle firme di ClamAV). I report restano sul tuo disco. L'intero comportamento è verificabile — è un unico file Bash leggibile.

In cosa è diverso da Lynis?

Lynis è un eccellente auditor puro — ti dice cosa non va. antivirus.sh ha opinioni precise e si sporca le mani: dà anche la caccia al malware attivo e corregge ciò che trova, in sicurezza, con rollback. Eseguili entrambi, se vuoi; non sono quasi mai in disaccordo.

Come lo eseguo regolarmente?

Aggiungi una riga di cron (vedi Utilizzo) con --audit --no-color --report. Gli exit code 0/1/2 rendono banale generare avvisi da CI, dalle mail di cron o da qualsiasi sistema di monitoraggio.
