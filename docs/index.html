Why antivirus.sh

Most Linux machines are servers that get compromised through the same boring doors: exposed ports, password SSH, forgotten cron jobs, unpatched packages. antivirus.sh closes those doors in minutes — and looks for anyone who already walked in.

📦

Single file, zero dependencies

One Bash script. No Python, no agents, no daemons, nothing to compile. Download, run, delete if you like — it leaves a report and (optionally) a hardened system behind.

🐧

Any Ubuntu. Any Linux.

Guaranteed on every Ubuntu release — from ancient LTS to the latest. Best-effort support for Debian, RHEL, CentOS, Alma, Rocky, Fedora, Arch, openSUSE, Alpine: every probe degrades gracefully.

🔍

Finds real-world malware

Crypto-miners, reverse shells, cron droppers, SSH-key backdoors, LD_PRELOAD rootkits, fileless memfd payloads, hidden processes — the things actually found on hacked servers.

🧯

Fixes, not just findings

Every problem comes with a fix it can apply: interactively, automatically with --fix, or as a full new-VM setup with --harden. Or stay 100% read-only with --audit.

🪢

Won't lock you out

Firewall enablement pre-allows your SSH ports. Key-only SSH is refused until a key actually exists. New sshd configs are validated before reload. Everything is backed up and reversible with --rollback.

🤝

Plays well with the big guns

Detects and orchestrates ClamAV, rkhunter and chkrootkit when present — one flag installs them. Use antivirus.sh as the fast first line and the glue around deep scanners.

The full check catalog

More than 70 individual checks across six areas. Every finding is classified OK / INFO / WARN / CRIT and feeds a final 0–100 security score.

01 Malware & rootkits

  • known miner & malware process names (xmrig, kinsing, kdevtmpfsi…) and mining-pool command lines
  • processes running from /tmp, /var/tmp, /dev/shm
  • processes whose binary was deleted; fileless memfd executables
  • hidden-process detection (readdir-hiding kernel rootkit test)
  • kernel modules matching known rootkits (Diamorphine, Reptile…)
  • /etc/ld.so.preload & global LD_PRELOAD userland rootkits
  • hidden executables in tmp dirs and /dev
  • reverse-shell / dropper / miner patterns in scripts up to 2 MB
  • core binary integrity vs package checksums (dpkg -V / rpm -V)
  • ClamAV, rkhunter, chkrootkit orchestration (optional)

02 Persistence & backdoors

  • system & user crontabs, cron.d, hourly/daily/weekly/monthly, at jobs
  • custom systemd units executing from tmp or matching attack patterns
  • rc.local, init scripts, udev rules, MOTD scripts
  • shell startup files of every user (.bashrc, .profile, .zshrc…)
  • authorized_keys audit for every account — flags SSH keys on system accounts
  • extra UID-0 accounts, NOPASSWD sudo rules, legacy .rhosts trust files
  • third-party APT repositories review

03 Network exposure

  • every listening TCP/UDP port with owning process and bind address
  • exposure classification: loopback-only vs world-reachable
  • dangerous services flagged: Telnet, FTP, r-services, SMB, RDP, VNC, rsync, NFS, rpcbind
  • unauthenticated-by-default stores: Redis, MongoDB, Elasticsearch, memcached
  • Docker API on :2375 (instant remote root) detection
  • established connections to mining-pool / IRC-botnet ports
  • promiscuous interfaces, ARP-spoofing indicators
  • DNS resolver review and /etc/hosts hijack detection

04 SSH & accounts

  • root login, password authentication, empty passwords, X11 forwarding
  • MaxAuthTries, login grace, keep-alives, host-based auth, rhosts
  • fail2ban presence with auto-setup of an sshd jail
  • brute-force evidence in btmp failed-login records
  • empty-password accounts (with one-key lock), duplicate UIDs
  • system accounts with login shells, sudo group membership review
  • password aging policy and pam_pwquality strength enforcement

05 Kernel & filesystem

  • 16 sysctl hardening keys: ASLR, syncookies, redirects, source routing, rp_filter, ptrace scope, dmesg/kptr restriction, protected links…
  • permissions on shadow, sudoers, SSH host keys, crontab, GRUB config
  • SUID/SGID audit against a curated whitelist + package-ownership proof
  • world-writable files and unsticky world-writable directories
  • unowned files, immutable-flag (chattr +i) malware locks
  • /dev/shm & /tmp mount flags (noexec, nosuid, nodev)
  • core-dump restrictions and SUID dump prevention

06 Platform & services

  • pending security updates, unattended-upgrades, EOL release detection
  • reboot-required detection after kernel patches
  • AppArmor / SELinux mandatory access control state
  • auditd forensic trail, persistent journald logging
  • NTP time synchronization (TLS & forensics depend on it)
  • CPU vulnerability mitigations (Meltdown/Spectre family)
  • Docker: socket permissions, privileged containers, docker-group members
  • Secure Boot state, GRUB password recommendation

Install & first run

No installation in the classic sense — it is just a script. Three commands and you have a full report.

download
curl -fsSL https://raw.githubusercontent.com/TARGET_PLEVEHOLDER/antivirus.sh/main/antivirus.sh -o antivirus.sh
1 — safe read-only audit (changes nothing)
sudo bash antivirus.sh --audit
2 — interactive: review and confirm each fix
sudo bash antivirus.sh
3 — or harden a fresh VM end-to-end
sudo bash antivirus.sh --harden

Modes & options

One script, four ways to use it: audit, interactive, automatic, full hardening. Plus focused scopes for scripting and cron.

CommandWhat happens
sudo bash antivirus.shInteractive mode — every finding is shown, every fix individually confirmed. Recommended first run.
--auditReport only. Guaranteed zero changes — safe on production. Exit code 0/1/2 = clean/warnings/critical, perfect for cron and CI.
--fixApply all safe fixes automatically. Risky fixes (firewall, SSH auth) still require explicit confirmation.
--hardenEverything --fix does, plus the full new-VM kit: firewall, fail2ban, unattended-upgrades, sysctl set, auditd, password policy, banners, core-dump & shm lockdown, and an offer to create a proper admin user.
--create-userGuided creation of a sudo user: strong generated password, SSH key install, optional switch to key-only SSH — with a reminder to test before closing your session.
--scan /var/wwwMalware-scan a specific directory (web roots are a favourite hiding place).
--network / --system / --malwareRun a single area only.
--quickSkip the slow parts (full-disk sweeps, ClamAV) — finishes in seconds.
--fullDeepest mode: whole filesystem scan, every installed package verified.

--install-toolsInstall ClamAV, rkhunter and chkrootkit for signature-based depth.

Built so you cannot lock yourself out

Hardening tools have a bad habit of cutting the branch you are sitting on. antivirus.sh treats remote access as sacred:

  • Firewall enablement pre-allows SSH. Your SSH ports are detected three ways (sshd config, live sockets, current connection) and allowed before ufw enable runs.
  • Key-only SSH is refused without a key. PasswordAuthentication no is only applied if a sudo-capable user actually has an authorized_keys entry. Same for disabling root login.
  • sshd configs are validated before reload. Every change passes sshd -t; a failing config is restored automatically. Live SSH sessions are never dropped.
  • Everything is backed up. Each modified file is copied to a timestamped backup directory with a manifest; --rollback restores all of it, including disabling a firewall we enabled.
  • Quarantine, never delete. Suspicious files are moved to a quarantine directory and stripped of all permissions — evidence and reversibility preserved.
  • Risky fixes always ask. Even in --fix mode, anything that could affect remote access requires explicit confirmation (or the documented, deliberate --yes).

Recipe: brand-new VM in 3 minutes

You just got a fresh Ubuntu VM from your cloud provider, root password in hand. Here is the whole routine:

step 1 — fetch and harden
curl -fsSL https://raw.githubusercontent.com/TARGET_PLEVEHOLDER/antivirus.sh/main/antivirus.sh -o antivirus.sh
sudo bash antivirus.sh --harden

During the run it will: enable the firewall with SSH pre-allowed → install fail2ban & unattended-upgrades → apply 16 kernel hardening keys → tighten SSH (validated, session-safe) → set password policy → lock down /dev/shm, core dumps and idle shells → offer to create your personal sudo user with an SSH key → and only then, with your key in place, offer to switch SSH to key-only and disable root login. Test the new login in a second terminal — your current session stays alive the whole time.

FAQ

Is it a "real" antivirus like ClamAV?

It is a different layer. ClamAV matches millions of file signatures; antivirus.sh hunts the patterns that actually compromise Linux servers — miners, reverse shells, cron droppers, SSH backdoors, rootkit traces — and audits the configuration that let them in. They complement each other, which is why antivirus.sh detects ClamAV, uses it when present, and installs it for you with --install-tools.

Will it work on my ancient Ubuntu 14.04 / brand-new release?

Yes — that is a design constraint. Pure Bash, no compiled parts, every probe has fallbacks (ss/netstat, systemd/sysvinit, ufw/iptables) and skips gracefully with a note when a facility does not exist. The same script runs from Ubuntu 12.04-era systems to the latest release.

Is it safe to run on production?

--audit mode is strictly read-only and is the recommended first run anywhere. Modes that change things confirm each step (interactive) or restrict themselves to safe fixes (--fix), keep backups of every touched file, and provide --rollback.

Can it really not lock me out of SSH?

The dangerous transitions — enabling a firewall, disabling password auth, disabling root login — are guarded: SSH ports are pre-allowed, key presence is verified before key-only mode, configs are validated with sshd -t and auto-reverted on failure, and your current session is never dropped. You are still advised to test a new login in a second terminal; the script reminds you at exactly the right moment.

It found something CRITICAL. What now?

Read the finding — each one explains itself and prints the evidence. For malware findings, quarantine via the offered fix, then ask how it got there (exposed service? stolen key? old CMS?). Important: if a kernel rootkit is indicated (hidden processes, rootkit modules), no cleanup on the live system is trustworthy — rebuild from a clean image and restore data only.

Does it phone home or upload anything?

No. There is no telemetry, no uploads, no external calls except the ones you ask for (package installs, ClamAV signature updates). Reports stay on your disk. The entire behaviour is auditable — it is one readable Bash file.

How is it different from Lynis?

Lynis is a superb pure auditor — it tells you what is wrong. antivirus.sh is opinionated and hands-on: it also hunts active malware, and it fixes what it finds, safely, with rollback. Run both if you like; they disagree on almost nothing.

How do I run it regularly?

Add a cron line (see Usage) with --audit --no-color --report. Exit codes 0/1/2 make it trivial to alert from CI, cron mail or any monitoring system.
